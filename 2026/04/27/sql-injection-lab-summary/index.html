SQL 注入实验复盘：字符串拼接与参数化查询对照

这次实验主要围绕一个核心问题展开：

当程序把用户输入直接拼接进 SQL 语句时，用户输入就可能改变 SQL 原本的结构和逻辑，从而产生 SQL 注入风险。

为了更直观地理解这个问题，我在同一台 CentOS 虚拟机上分别搭建了两个版本：

版本 访问地址 核心特点
安全版 http://192.168.88.10:5000 使用参数化查询
不安全版 http://192.168.88.10:5001 使用字符串拼接 SQL

一、实验目标

本次实验主要验证两个问题：

  1. 为什么字符串拼接 SQL 会产生 SQL 注入风险
  2. 为什么参数化查询可以防止这类问题

最终目标不是单纯地“登录成功”，而是理解：

  • SQL 注入是如何发生的
  • 用户输入为什么会影响 SQL 语句结构
  • 参数化查询为什么能够阻断这类风险
