这次实验主要围绕一个核心问题展开：当程序把用户输入直接拼接进 SQL 语句时，用户输入就可能改变 SQL 原本的结构和逻辑，从而产生 SQL 注入风险。

为了更直观地理解这个问题，我在同一台 CentOS 虚拟机上分别搭建了两个版本：一个使用字符串拼接 SQL 的不安全版，另一个使用参数化查询的安全版。

一、实验目标

这次实验主要验证两个问题：

  1. 为什么字符串拼接 SQL 会产生 SQL 注入风险
  2. 为什么参数化查询可以防止这类问题

十二、最终结论

SQL 注入的根本原因，不是 MySQL 本身有问题，而是程序把用户输入直接拼接进 SQL 语句，导致用户输入有机会改变 SQL 的结构和逻辑。

正确的防护方式是使用参数化查询或预编译语句，让 SQL 结构和用户输入彻底分离。
